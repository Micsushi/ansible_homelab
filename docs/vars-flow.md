# Variables Flow — How Secrets and Config Work

This document explains exactly which files hold what, how they interact, and what you need to do before each stage.

---

## File roles at a glance

```
group_vars/all/
├── vars.yml            ← TRACKED TEMPLATE (committed to git, no real secrets)
├── vars_local.yml      ← GITIGNORED OVERRIDES (your real secrets, never committed)
├── vars_local.yml.example  ← reference for what vars_local.yml should contain
└── gui_apps.yml        ← GUI app lists (tracked; no secrets)

.generated-values.env   ← GITIGNORED (random secrets generated by scripts/generate-values.sh)
```

---

## What goes where

| Variable | File | Why |
|---|---|---|
| `username`, `puid`, `pgid`, `timezone`, `domain` | `vars.yml` | Not secret; safe to commit |
| `docker_dir`, `data_dir`, `deploy_*` flags | `vars.yml` | Defaults; safe to commit |
| `pihole_adlists`, `pihole_upstream_dns` | `vars.yml` | Config; safe to commit |
| `ip_address` | **`vars_local.yml`** | Identifies your LAN; keep private |
| `immich_postgres_password`, `immich_typesense_api_key` | **`vars_local.yml`** | Generated secret |
| `samba_password`, `pihole_password` | **`vars_local.yml`** | Generated secret |
| `immich_admin_email/name/password` | **`vars_local.yml`** | Your chosen admin credentials |
| `jellyfin_admin_password` | **`vars_local.yml`** | Your chosen admin password |
| `jwt_secret`, `authelia_sqlite_encryption_key` | **`vars_local.yml`** | Generated secret (Stage 3) |
| `vnc_password`, `coolify_postgres_password` | **`vars_local.yml`** | Generated secret (Stage 3) |
| Stage 3 Cloudflare/Authelia/Tailscale values | **`vars_local.yml`** | Must be set manually |

## How Ansible merges them

Ansible loads every YAML file under `group_vars/all/` in alphabetical order.
`vars_local.yml` sorts **after** `vars.yml`, so for the **same key**, `vars_local.yml` wins.

Example: `vars.yml` has `pihole_password: "<pihole_password>"`.
After you run `apply-generated-values.sh --stage 2`, `vars_local.yml` has `pihole_password: "realvalue"`.
Ansible uses **`realvalue`** — the placeholder in `vars.yml` is never seen.

## File lifecycle

### `.generated-values.env`
- Created by: `bash scripts/generate-values.sh`
- Rotated by: `bash scripts/generate-values.sh --overwrite`
- Contains: raw key=value pairs, all random
- **Never applied automatically** — you run `apply-generated-values.sh` to use it

### `group_vars/all/vars_local.yml`
- Created/updated by: `bash scripts/setup-local-vars.sh` or `apply-generated-values.sh`
- Contains: real secrets and private overrides only
- Backed up to `vars_local.yml.bak.<timestamp>` before each `apply` write
- **Gitignored** — stays on your control node only

### `group_vars/all/vars.yml`
- Created/updated by: you, in git
- Contains: defaults, placeholders (`<key_name>`), feature toggles
- **Tracked in git** — never put real secrets here

### `vars_local.yml.bak.<timestamp>`
- Created by: `apply-generated-values.sh` before each write
- Purpose: instant rollback (`bash scripts/apply-generated-values.sh --revert-last`)
- Gitignored

## Commands reference

```bash
# First-time setup (interactive — does everything)
bash scripts/setup-local-vars.sh

# Sanity-check vars_local.yml without printing secrets
bash scripts/check-vars-local.sh

# Rotate Stage 2 secrets only
bash scripts/generate-values.sh --overwrite
bash scripts/apply-generated-values.sh --stage 2

# Rotate Stage 3 secrets only
bash scripts/generate-values.sh --overwrite
bash scripts/apply-generated-values.sh --stage 3

# Rollback vars_local.yml to last backup
bash scripts/apply-generated-values.sh --revert-last

# Sync Windows repo → WSL (preserves vars_local.yml on WSL side)
.\scripts\sync-to-wsl.ps1    # from PowerShell
```
